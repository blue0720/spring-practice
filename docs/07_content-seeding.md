## 実装メモ

### 投入方法
`ContentSeeder`(`com.example.demo.config.ContentSeeder`、`ApplicationRunner`)がアプリ起動時に
`chapterテーブルが空の場合のみ`第1〜3章のコンテンツを投入する(冪等)。チケット06で実装した
`AdminChapterService`/`AdminNpcService`/`AdminDistrictService`/`AdminMemoryFragmentService` の
`save(Form)` を経由して作成するため、管理画面から手入力した場合と全く同じ入口・同じ手順で
データが構成される(「管理画面またはシードデータで投入する」という本チケットの要求を、
両者を同じService層に統一することで両立させた)。Flyway/Liquibase未導入(`docs/09_...md`参照)の
現状に合わせ、SQLではなくJavaコードでシードすることでエンティティの関連(Chapter→Npc→District の
生成済みIDの受け渡し)を型安全に扱える。

### 投入したコンテンツの対応
- 第1章「依頼」: 貧困街 = ノラ(靴職人、弱点=安心/苦手=恐怖)。既視感の伏線は、記憶断片プールの
  「愛情」タグの断片タイトルを「誰かを守るために差し出した、名もなき覚悟」とすることで表現した
  (MemoryFragmentは章に依存しない全章共通プールのため、専用データを追加するのではなく、
  既存8種の一つに伏線の意味を持たせる形にした)。
- 第2章「値札のない記憶」: ギルド本部 = ヴァイス(記録官、弱点=誇り/苦手=恥)、裏路地 = レイム
  (記憶ブローカー、弱点=愛情/苦手=怒り。中盤の山場の警告台詞はレイムの`introText`に含めた)、
  テオドール(記憶保管人、区画には属さず章のみ紐付け)。
- 第3章「代償と選択」: セレーネ(最終交渉相手)を投入。`weakTag`/`hateTag`は要件定義書の
  未決事項1(相性表ロジック流用か専用画面か)が未確定のため意図的に`null`のまま(カラムは
  nullable)。妹の現在・セレーネの正体開示などのエンディング分岐テキストは、専用エンティティを
  新設せず`Chapter`(第3章)の`description`に格納した(既存エンティティ構成を変えないため)。
- 脇役: 「常連の老人」を第1章に投入。会話ノルマがないモブという設定に合わせ、反応テキストは
  1件も作成していない。
- テオドールの相性(弱点=喜び/苦手=悲しみ = 「喪失を抱えた市民」タイプ)は、企画書の相性表4種の
  うち唯一具体的なNPCが割り当てられていなかった枠を、彼の「他者の喪失を記録・弔う」という
  人物像に基づいて割り当てた(企画書に直接の明記はないための設計判断)。

### 相性表の整合性確認
`ContentSeederTest`(`mainNpcAffinityMatchesGameLogicAffinityTable`)で、ノラ/ヴァイス/レイム/
テオドールの`weakTag`/`hateTag`が `static/js/game-data.js` の `AFFINITY_TABLE`
(孤独な貧困層・ギルド幹部権力者・裏市場の人間・喪失を抱えた市民の4種)と一致することを
自動テストで固定した。あわせて、記憶断片プールが8感情タグを過不足なくカバーしていること、
セレーネ/モブが意図通り特別扱い(タグ未設定・反応テキストなし)であることも検証している。

### 既存テストとの衝突回避
シードした「ノラ」「第1章」等がDBに恒久的に残るため、`AdminNpcControllerTest`・
`AdminChapterControllerTest`が同名/同番号のテストデータを作成すると一意制約
(`chapter.number`)や`findByName`の複数件ヒットで失敗する。テスト側のフィクスチャ名・章番号を
シード内容と衝突しない値(「検証用テストNPC」・章番号99)に変更して回避した。
